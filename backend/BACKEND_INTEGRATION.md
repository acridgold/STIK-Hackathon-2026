# Руководство по интеграции бэкенда

Этот документ объясняет, как фронтенд подключается к бэкенду.

## Текущий статус

В настоящее время фронтенд использует **Zustand store с localStorage** в качестве временного слоя данных (MVP версия). Когда вы будете готовы подключиться к реальному бэкенду, вам нужно:

1. Реализовать конечные точки API бэкенда (рекомендуется Go + Gin)
2. Обновить слой API сервиса фронтенда
3. Настроить базовый URL API

## Спецификация API бэкенда

Фронтенд ожидает RESTful API со следующими конечными точками:

### Базовый URL
- По умолчанию: `http://localhost:8080/api`
- Настраивается через переменную окружения: `VITE_API_URL`

### Конечные точки

#### Компании
- `GET /api/companies` - Получить все компании
- `POST /api/companies` - Создать компанию
- `PUT /api/companies/:id` - Обновить компанию
- `DELETE /api/companies/:id` - Удалить компанию

#### Курсы
- `GET /api/courses` - Получить все курсы
- `POST /api/courses` - Создать курс
- `PUT /api/courses/:id` - Обновить курс
- `DELETE /api/courses/:id` - Удалить курс
- `GET /api/courses/:id/price-history` - Получить историю цен

#### Сотрудники
- `GET /api/employees` - Получить всех сотрудников
- `POST /api/employees` - Создать сотрудника
- `PUT /api/employees/:id` - Обновить сотрудника
- `DELETE /api/employees/:id` - Удалить сотрудника

#### Группы
- `GET /api/groups` - Получить все группы
- `GET /api/groups/:id` - Получить конкретную группу
- `POST /api/groups` - Создать группу
- `PUT /api/groups/:id` - Обновить группу
- `DELETE /api/groups/:id` - Удалить группу
- `POST /api/groups/:id/participants` - Добавить участника
- `DELETE /api/groups/:id/participants/:pid` - Удалить участника
- `PATCH /api/groups/:id/participants/:pid/progress` - Обновить прогресс

#### Спецификации
- `GET /api/specifications` - Получить все спецификации
- `GET /api/specifications/:id` - Получить конкретную спецификацию
- `POST /api/specifications` - Создать спецификацию
- `PUT /api/specifications/:id` - Обновить спецификацию
- `DELETE /api/specifications/:id` - Удалить спецификацию

#### Импорт XML
- `POST /api/import/xml` - Импортировать XML данные (multipart/form-data)

#### Аналитика
- `GET /api/analytics/companies` - Сводка по компаниям
- `GET /api/analytics/schedule-conflicts` - Конфликты расписания

## Слой API сервиса фронтенда

Сервис API фронтенда находится в `frontend/src/api.js`. Этот файл содержит:

1. **Конфигурация BASE_URL**: `const BASE_URL = import.meta.env.VITE_API_URL || 'http://localhost:8080/api'`
2. **Вспомогательная функция запросов**: Обрабатывает HTTP запросы с типом содержимого JSON
3. **Методы API**: Организованы по типам сущностей (компании, курсы, сотрудники и т.д.)

## Как подключиться к бэкенду

### Шаг 1: Реализовать API бэкенда

Реализуйте конечные точки API согласно спецификации выше. Рекомендуемый стек:
- **Язык**: Go
- **Фреймворк**: Gin
- **База данных**: PostgreSQL/MySQL

### Шаг 2: Настроить фронтенд

Обновите базовый URL API в вашем фронтенд окружении:

```bash
# В вашем .env файле (директория фронтенда)
VITE_API_URL=http://your-backend-url/api
```

Или установите его во время сборки:

```bash
# Для разработки
VITE_API_URL=http://localhost:8080/api npm run dev

# Для продакшн сборки
VITE_API_URL=https://your-production-api.com/api npm run build
```

### Шаг 3: Обновить API сервис (опционально)

Если ваш бэкенд использует другие:
- Пути URL
- Форматы запросов/ответов
- Методы аутентификации

Обновите файл `frontend/src/api.js` соответственно.

## Формат запросов/ответов

### Заголовки запросов
- `Content-Type: application/json` (для JSON конечных точек)
- `multipart/form-data` (для загрузки файлов)

### Успешный ответ
- Статус: 200 OK
- Content-Type: application/json
- Тело: JSON объект с данными

### Ошибка ответа
- Статус: 4xx/5xx
- Content-Type: application/json
- Тело: `{ "message": "Описание ошибки" }`

## Аутентификация

В настоящее время фронтенд не реализует аутентификацию. Если ваш бэкенд требует аутентификации:

1. Добавьте заголовки аутентификации в функцию `request()` в `api.js`
2. Реализуйте потоки входа/выхода
3. Добавьте логику хранения и обновления токенов

## Рабочий процесс разработки

### Локальная разработка
1. Запустите бэкенд: `go run main.go` (или вашу команду бэкенда)
2. Запустите фронтенд: `npm run dev`
3. Фронтенд будет проксировать API запросы на `http://localhost:8080/api`

### Развертывание в продакшн
1. Соберите фронтенд: `npm run build`
2. Разверните бэкенд на вашем сервере
3. Настройте фронтенд для указания на продакшн URL API
4. Разверните статические файлы фронтенда

## Устранение неполадок

**Проблема**: Запросы API завершаются с ошибками CORS
**Решение**: Убедитесь, что на вашем бэкенде включено CORS middleware:
```go
// Пример для Gin
router.Use(cors.Default())
```

**Проблема**: Фронтенд не может подключиться к бэкенду
**Решение**:
1. Проверьте, что бэкенд запущен
2. Проверьте конфигурацию базового URL API
3. Протестируйте конечные точки с помощью Postman/curl
4. Проверьте ошибки в консоли браузера

**Проблема**: Данные не обновляются в UI
**Решение**: Фронтенд может использовать кэшированные данные из Zustand. Очистите хранилище или реализуйте правильную инвалидацию кэша.

## Пример реализации бэкенда (Go + Gin)

```go
package main

import (
    "github.com/gin-gonic/gin"
    "github.com/gin-contrib/cors"
)

func main() {
    router := gin.Default()
    
    // Включить CORS
    router.Use(cors.Default())
    
    // Маршруты API
    api := router.Group("/api") {
        // Компании
        api.GET("/companies", getCompanies)
        api.POST("/companies", createCompany)
        // ... другие конечные точки
    }
    
    router.Run(":8080")
}
```

## Миграция с localStorage на реальный бэкенд

Когда вы будете готовы перейти с localStorage на реальный бэкенд:

1. **Обновите API сервис**: Замените вызовы Zustand store на реальные API вызовы в `api.js`
2. **Обновите компоненты**: Замените прямое использование хранилища на API вызовы
3. **Тщательно протестируйте**: Убедитесь, что все CRUD операции работают с бэкендом
4. **Разверните**: Сначала разверните бэкенд, затем обновите фронтенд

Фронтенд разработан чтобы быть независимым от бэкенда, поэтому вы можете реализовать бэкенд на любой технологии, если он следует API контракту.
